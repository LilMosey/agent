# 企业知识库文件管理安装使用说明

## 这个工程是干什么的

本工程是一个企业知识库文件管理系统的第一版，实现“知识库空间”和“知识库文件”的基础管理能力。

当前版本支持：

- 创建、查看、编辑、删除知识库。
- 在某个知识库下上传文件。
- 按文件名关键字和文件状态筛选文件。
- 查看文件详情。
- 下载原始文件。
- 删除文件，并同步删除对象存储中的原始文件。
- 防止同一知识库下上传同名文件。
- 防止知识库下还有文件时直接删除知识库。

当前版本不包含：

- 登录和权限控制。
- RAG 文档解析。
- 向量写入和向量检索。
- Agent 应用能力。

Milvus、Attu 等组件已经在本地 Docker 环境中预留，但第一版文件管理流程主要依赖 MySQL 和 MinIO。

## 技术栈

后端：

- Java 21
- Spring Boot 3.5.0
- Maven 多模块
- MyBatis-Plus
- MySQL
- MinIO

前端：

- React 19
- Vite 7
- TypeScript
- Ant Design
- Axios
- lucide-react

本地基础设施：

- MySQL：保存知识库和文件元数据。
- MinIO：保存上传的原始文件。
- Milvus：后续 RAG 阶段使用，当前版本不强依赖。
- Attu：Milvus 管理界面，当前版本不强依赖。

## 工程结构

```text
.
├── backend                         # 后端 Maven 多模块工程
│   ├── kb-domain                   # 领域模型和领域规则
│   ├── kb-application              # 应用服务和端口接口
│   ├── kb-api                      # REST API、DTO、异常处理
│   ├── kb-infrastructure           # MySQL、MyBatis、MinIO 等基础设施实现
│   └── kb-bootstrap                # Spring Boot 启动模块
├── frontend                        # 前端 React/Vite 工程
├── docs                            # 项目设计和说明文档
└── docker-compose.yml              # 本地 MySQL、MinIO、Milvus、Attu 环境
```

后端模块职责：

- `kb-domain`：定义 `KnowledgeBase`、`KnowledgeFile`、`FileType`、`FileStatus` 等领域对象，并通过 `FileTypePolicy` 限制允许上传的文件类型。
- `kb-application`：封装知识库和文件的业务用例，例如创建知识库、上传文件、下载文件、删除文件。
- `kb-api`：提供 HTTP 接口，统一返回 `ApiResponse`，并处理业务异常和参数校验异常。
- `kb-infrastructure`：实现数据库持久化和 MinIO 对象存储。
- `kb-bootstrap`：组装 Spring Bean，启动后端应用。

## 启动本地依赖

在项目根目录执行：

```bash
docker compose up -d
```

该命令会启动：

| 服务 | 地址 | 说明 |
| --- | --- | --- |
| MySQL | `localhost:3306` | 元数据数据库 |
| MinIO API | `http://localhost:9000` | 文件对象存储 API |
| MinIO 控制台 | `http://localhost:9001` | 文件对象存储管理页面 |
| Milvus | `localhost:19530` | 后续向量检索预留 |
| Attu | `http://localhost:8000` | Milvus 管理页面 |

MinIO 控制台账号：

```text
用户名：minioadmin
密码：minioadmin123
```

MySQL 默认配置：

```text
数据库：enterprise_kb
用户名：enterprise_kb
密码：enterprise_kb123456
```

`docker-compose.yml` 会把 `backend/kb-infrastructure/src/main/resources/db/schema.sql` 挂载到 MySQL 初始化目录，用于创建 `knowledge_base` 和 `knowledge_file` 两张表。

如果容器已经存在且数据库卷中已经初始化过，修改 `schema.sql` 后不会自动重新执行初始化脚本。需要自行迁移数据库，或在确认可以清空本地数据后重建 Docker 卷。

## 启动后端

进入后端目录：

```bash
cd backend
```

先编译：

```bash
mvn -q -DskipTests compile
```

启动 Spring Boot：

```bash
mvn spring-boot:run -pl kb-bootstrap
```

后端默认监听：

```text
http://localhost:8080
```

关键配置位于：

```text
backend/kb-bootstrap/src/main/resources/application.yml
```

默认后端配置会连接：

- MySQL：`jdbc:mysql://localhost:3306/enterprise_kb`
- MinIO：`http://localhost:9000`
- MinIO bucket：`enterprise-kb-files`

## 启动前端

新开一个终端，进入前端目录：

```bash
cd frontend
```

安装依赖：

```bash
npm install
```

启动开发服务器：

```bash
npm run start
```

前端默认访问地址：

```text
http://localhost:5173
```

Vite 已配置代理：

```text
/api -> http://localhost:8080
```

所以前端代码中只需要请求 `/api/...`，开发环境会自动转发到后端。

## 前端页面怎么用

1. 打开 `http://localhost:5173`。
2. 点击左侧“新建”，创建一个知识库。
3. 选择知识库后，点击右上角“上传文件”。
4. 上传 Word、Markdown 或 TXT 文件。
5. 在文件列表中可以进行：
   - 搜索文件名。
   - 按状态筛选。
   - 查看详情。
   - 下载文件。
   - 删除文件。
6. 如果要删除知识库，需要先删除该知识库下的全部文件。

当前允许上传的文件扩展名：

```text
.doc
.docx
.md
.markdown
.txt
```

当前不支持上传 PDF 文件。上传 PDF 会返回“当前版本不支持上传 PDF 文件。”。

## 后端接口概览

所有业务接口统一以 `/api` 开头。

知识库接口：

| 方法 | 地址 | 说明 |
| --- | --- | --- |
| `POST` | `/api/knowledge-bases` | 创建知识库 |
| `GET` | `/api/knowledge-bases` | 获取知识库列表 |
| `GET` | `/api/knowledge-bases/{id}` | 获取知识库详情 |
| `PUT` | `/api/knowledge-bases/{id}` | 更新知识库 |
| `DELETE` | `/api/knowledge-bases/{id}` | 删除知识库 |

文件接口：

| 方法 | 地址 | 说明 |
| --- | --- | --- |
| `POST` | `/api/knowledge-bases/{kbId}/files` | 上传文件 |
| `GET` | `/api/knowledge-bases/{kbId}/files` | 查询文件列表 |
| `GET` | `/api/knowledge-bases/{kbId}/files/{fileId}` | 获取文件详情 |
| `GET` | `/api/knowledge-bases/{kbId}/files/{fileId}/download` | 下载文件 |
| `DELETE` | `/api/knowledge-bases/{kbId}/files/{fileId}` | 删除文件 |

文件列表支持查询参数：

| 参数 | 说明 |
| --- | --- |
| `keyword` | 按文件名关键字搜索 |
| `status` | 按文件状态过滤 |
| `page` | 页码，默认 `1` |
| `size` | 每页数量，默认 `20` |

文件状态包括：

```text
UPLOADED
PENDING_PARSE
PARSING
PARSE_FAILED
READY
DISABLED
```

第一版上传完成后的默认状态是 `UPLOADED`。

## curl 示例

创建知识库：

```bash
curl -X POST http://localhost:8080/api/knowledge-bases \
  -H "Content-Type: application/json" \
  -d '{"name":"产品资料库","description":"存放产品文档和说明"}'
```

查询知识库列表：

```bash
curl http://localhost:8080/api/knowledge-bases
```

上传文件：

```bash
curl -X POST http://localhost:8080/api/knowledge-bases/1/files \
  -F "file=@/path/to/example.txt"
```

查询文件列表：

```bash
curl "http://localhost:8080/api/knowledge-bases/1/files?keyword=example&page=1&size=20"
```

下载文件：

```bash
curl -L -o example.txt \
  http://localhost:8080/api/knowledge-bases/1/files/1/download
```

删除文件：

```bash
curl -X DELETE http://localhost:8080/api/knowledge-bases/1/files/1
```

删除知识库：

```bash
curl -X DELETE http://localhost:8080/api/knowledge-bases/1
```

## 推荐验证流程

1. 启动 Docker 依赖。
2. 启动后端。
3. 启动前端。
4. 打开前端页面。
5. 创建一个知识库。
6. 上传一个 `.txt` 文件。
7. 确认文件列表出现该文件，状态为 `UPLOADED`。
8. 上传同名文件，确认系统提示同名文件已存在。
9. 查看文件详情，确认文件名、大小、类型、状态、MinIO bucket 和 object key 正确。
10. 下载文件，确认内容正确。
11. 尝试在文件未清空时删除知识库，确认后端拒绝删除。
12. 删除文件后再删除知识库，确认删除成功。

## 常见问题

### 前端页面请求失败

先确认后端是否已经启动在 `8080` 端口：

```bash
curl http://localhost:8080/api/knowledge-bases
```

再确认前端开发服务器是否启动在 `5173` 端口。

### 后端启动后数据库连接失败

检查 MySQL 容器是否正常：

```bash
docker compose ps
```

确认 `application.yml` 中的数据库地址、用户名、密码和 `docker-compose.yml` 一致。

### 文件上传失败

优先检查：

- 文件扩展名是否为 `.doc`、`.docx`、`.md`、`.markdown` 或 `.txt`。
- 同一知识库下是否已经存在同名文件。
- MinIO 容器是否正常运行。
- MinIO 中是否存在 `enterprise-kb-files` bucket。

### 删除知识库失败

当前业务规则要求先删除知识库下所有文件，再删除知识库。文件未清空时删除知识库会返回：

```text
知识库下仍有文件，请先删除文件。
```

### Milvus 和 Attu 是否必须启动

第一版文件管理不强依赖 Milvus 和 Attu。它们是为后续 RAG、向量写入和检索能力预留的本地组件。
